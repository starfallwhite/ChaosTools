二、重新定义 MCP 和 Skill（必须统一）
✅ MCP（Model Context Protocol）
✔ 正确定义
MCP = 标准化工具调用协议 + 工具注册 + 调用路由
✔ 职责
1️⃣ 向 LLM 暴露工具（tool schema）
2️⃣ 接收 LLM 的 tool call
3️⃣ 调度对应 skill
4️⃣ 返回结构化结果
❌ 不再负责
✘ 不负责复杂 workflow DAG（由 LLM 决策）
✘ 不负责业务逻辑
✅ Skill（核心计算单元）
✔ 定义
Skill = 可被 LLM 调用的原子工具
✔ 示例（你系统中的）
chaos_simulate      → 模型仿真
fft_analysis        → 频谱分析
lyapunov_calculate  → 李雅普诺夫指数
phase_reconstruct   → 相空间

六、必须补充的关键设计（否则会崩）
⚠️ 1. Tool Schema（最关键）

LLM 是否能正确调用工具，取决于这个：

✅ chaos_simulate schema
{
  "name": "chaos_simulate",
  "description": "Simulate a chaotic system and return time series",
  "parameters": {
    "type": "object",
    "properties": {
      "model": {
        "type": "string",
        "enum": ["electrooptic_feedback"]
      },
      "params": {
        "type": "object"
      }
    },
    "required": ["model", "params"]
  }
}

👉 这一步非常关键，否则：

LLM会乱调用参数（常见坑）
⚠️ 2. 数据必须“LLM友好”

LLM不能处理：

巨大数组（1e6长度）
✅ 解决方案
{
  "summary": {
    "mean": ...,
    "std": ...
  },
  "data_preview": [...100个点...],
  "full_data_ref": "cache_id"
}
⚠️ 3. 状态管理（关键）

LLM是“无状态”的，但你的系统不是。

必须支持：
session memory

例如：

用户 → simulate
下一步 → fft（用的是上一步结果）

👉 MCP需要支持：

context_id
